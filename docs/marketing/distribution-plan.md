# 유통 액션 플랜 (Distribution) — 첫 사용자 10명

> 작성 2026-07-03. 전제: `npm run funnel` 실측상 **외부 유입 ≈ 0**(30일 makecc.vercel.app 실방문 1건, 나머지 본인 트래픽). 병목은 기능이 아니라 **사람**. 이 문서는 "코드 그만, 유통 시작"의 실행 계획.
> 에셋: `docs/marketing/launch-posts.md`(긱뉴스·PH·커뮤니티 초안) · `docs/marketing/demo.gif` · `scripts/capture-demo-gif.mjs`.

## 성공 지표 (허영 지표 금지)
업보트·조회수 말고 **퍼널 실전환**으로 판단한다. `npm run funnel` / `/admin/analytics`로 측정:
- **1차 목표(2주)**: 외부 세션발(源) `upload_started` **≥ 20**, `job_finished` **≥ 15**, `srt_downloaded` **≥ 8**.
- 관찰 축: page_view→upload_started(랜딩 설득력) · upload→finished(처리 신뢰성) · finished→srt_downloaded(결과 만족).
- 유입 경로(referrer)로 어느 채널이 실제 전환되는지 분리.

## 🔴 런칭 전 필수 체크 (하나라도 빠지면 첫인상 망함)
1. **워커 상시 가동** — 런칭 시간대 내내 GPU PC에서 `npm run worker:poll`. **워커 꺼지면 업로드가 queued에서 멈춰** = 방문자가 "고장난 서비스"로 인식(최악). 유통의 유일한 하드 블로커.
2. **직전 E2E 스모크** — 게스트로 실제 업로드→자막→SRT 다운로드 1회 통과 확인(prod).
3. **분석 살아있는지** — `/admin/analytics` 접속 + 테스트 page_view 잡히는지.
4. **첫인상 라인** — 랜딩 첫 화면 문구/데모 GIF가 "이게 뭔지 5초 안에" 전달되는지 최종 점검.
5. **용량 각오** — 워커 GPU 1대라 잡은 **직렬 처리**(1분 영상 ~1.5분). 소규모 런칭엔 충분하지만 버스트 오면 대기 생김. 게스트 캡 존재.

## Phase 1 — 긱뉴스 (news.hada.io) · **좁게 시작**
왜 먼저: 한국 개발자 커뮤니티라 self-host Whisper·잡큐 같은 기술 서사가 먹히고, 피드백 질이 높고, 관대함. 소규모라 워커 1대로 감당 가능.
- [ ] 워커 켜고 스모크 통과
- [ ] `launch-posts.md` §1 "Show GN" 글 + **데모 GIF/스크린샷** 첨부
- [ ] 게시 후 **2~3시간 실시간 대응**: 댓글 피드백 즉답, 버그 나오면 바로 기록(`docs/inquiries/`)
- [ ] 그날 `npm run funnel`로 첫 코호트 확인 — **page_view는 느는데 upload가 0이면 랜딩 설득 문제**, upload는 있는데 finished가 낮으면 처리/워커 문제
- **판단 게이트**: 실 upload가 몇 건이라도 나오면 → Phase 2. 방문만 있고 upload 0이면 → 랜딩/온보딩부터 고치고 재시도(확산 미루기).

## Phase 2 — 커뮤니티 확산 (Phase 1 신호 확인 후)
대상: 아카라이브 유튜버 채널 / 클리앙 / 디시 영상편집 갤 등. `launch-posts.md` §3 짧은 톤.
- [ ] **커뮤니티별 자기홍보 규칙 먼저 확인**(안 그러면 밴). 각 채널 문화에 맞춰 톤 조정.
- [ ] 한 번에 다 뿌리지 말고 **1~2곳씩** — 워커 용량·피드백 대응 여력 고려.
- [ ] 유입 경로별 전환 분리 관찰 → 전환 좋은 채널에 집중.

## 반복 루프 (매 Phase 후)
1. `npm run funnel [일수]`로 이탈 구간 특정.
2. 가장 큰 이탈 1곳만 고친다(랜딩 문구 / 업로드 UX / 처리 속도 안내 / SRT 유도).
3. 피드백은 `docs/inquiries/`에 건별 정리.
4. 다시 측정 → 개선 → 소규모 재노출.
